# CCNA-LABS

## Lab 16 – OSPFv2 Part 3 — DR/BDR, Network Types & Troubleshooting

### Objective

Configure a serial connection between R1 and R2 and enable OSPF. Troubleshoot three pre-existing OSPF misconfigurations involving network type mismatch, Hello/Dead timer mismatch, and a missing default route. Verify the LSDB and identify LSA types present.

---

### Topology

- 5x Routers (R1, R2, R3, R4, R5)
- 2x Cisco 2960-24TT Switches (SW1, SW2, SW3)
- 2x PCs (PC1, PC2)
- 1x ISP Router (ISPR1)
- R1 ↔ R2: Serial (192.168.12.0/30) — newly added connection
- R2 ↔ R4/R5: GigabitEthernet (192.168.245.0/29)
- R3 ↔ R4: GigabitEthernet (192.168.34.0/30)
- R5 ↔ ISPR1: GigabitEthernet (203.0.113.0/30)
- R1 ↔ SW1 ↔ PC1: GigabitEthernet (10.0.1.0/24)
- R3 ↔ SW2 ↔ PC2: GigabitEthernet (10.0.2.0/24)
- OSPF Area: 0 (single area)

---

### Step 1 — Configure Serial Connection and OSPF (R1 and R2)

#### Verify DCE Side

```
show controllers s0/0/0
! Look for "DCE" in output — DCE side must set clock rate
```

#### R1 Configuration (DCE side)

```
interface s0/0/0
 clock rate 128000
 ip address 192.168.12.1 255.255.255.252
 no shutdown
 ip ospf 1 area 0

interface g0/0
 ip ospf 1 area 0
```

#### R2 Configuration (DTE side)

```
interface s0/0/0
 ip address 192.168.12.2 255.255.255.252
 no shutdown
 ip ospf 1 area 0
```

---

### Step 2 — Troubleshoot: Only R3 Has Route to 10.0.2.0/24

**Problem:** R3's G0/1 was configured with `ip ospf network point-to-point` while R4's G0/1 was using the default broadcast network type. When OSPF network types don't match, neighbors can appear FULL but routes are not exchanged properly.

**Diagnosis:**
```
show ip ospf interface g0/1   ! On both R3 and R4 — compare Network Type field
```

**Fix (on R3):**
```
interface g0/1
 no ip ospf network point-to-point
```

**Result:** Adjacency dropped and reformed. 10.0.2.0/24 appeared in R4's routing table.

---

### Step 3 — Troubleshoot: R2 and R4 Won't Become OSPF Neighbors with R5

**Problem:** R5's G0/0 had non-default Hello/Dead timers (Hello=5, Dead=20) while R2 and R4 used the defaults (Hello=10, Dead=40). Hello and Dead timers must match for OSPF neighbors to form.

**Diagnosis:**
```
show ip ospf interface g0/0   ! On R5 — check Timer intervals
```

**Fix (on R5):**
```
interface g0/0
 no ip ospf hello-interval
 no ip ospf dead-interval
```

**Result:** R5 immediately formed FULL adjacencies with R2 and R4.

---

### Step 4 — Troubleshoot: PC1 and PC2 Can't Ping 8.8.8.8

**Problem:** R5 had `default-information originate` configured but no actual default route in its routing table. Without a default route, OSPF has nothing to advertise as a Type 5 AS-External LSA.

**Diagnosis:**
```
show ip route   ! On R5 — "Gateway of last resort is not set" confirms no default route
```

**Fix (on R5):**
```
ip route 0.0.0.0 0.0.0.0 203.0.113.2

router ospf 1
 default-information originate
```

**Result:** R1 received `O*E2 0.0.0.0/0` via R2. PC1 successfully pinged 8.8.8.8.

---

### Step 5 — Examine the LSDB

```
show ip ospf database
```

| LSA Type | Name | Count | Generated By |
|----------|------|-------|-------------|
| Type 1 | Router LSA | 5 | Every router (R1, R2, R3, R4, R5) |
| Type 2 | Network LSA | 2 | DR of each multi-access segment |
| Type 5 | AS-External LSA | 1 | R5 (ASBR) — advertising default route to Internet |

### Troubleshooting Encountered

| Issue | Cause | Resolution |
|-------|-------|------------|
| Serial interface stayed down after `no shutdown` on R1 | R2's end was not yet configured | Configured R2's S0/0/0 IP and `no shutdown` — link came up |
| 10.0.2.0/24 missing from routing tables | R3 G0/1 = point-to-point, R4 G0/1 = broadcast — network type mismatch | Removed `ip ospf network point-to-point` from R3 G0/1 |
| R5 had no OSPF neighbors | Hello=5/Dead=20 on R5 vs Hello=10/Dead=40 default on R2 and R4 | Reset timers on R5 with `no ip ospf hello-interval` and `no ip ospf dead-interval` |
| Default route not advertised by R5 | `default-information originate` was configured but no static default route existed | Added `ip route 0.0.0.0 0.0.0.0 203.0.113.2` on R5 |

---

### Key Observations

- Serial interfaces default to the **point-to-point** OSPF network type — no DR/BDR election occurs
- OSPF network type mismatch is tricky — neighbors can reach FULL state but routes won't appear in the routing table
- Hello/Dead timer mismatch prevents neighbor formation entirely — always check timers when adjacency won't form
- `default-information originate` requires an actual default route to exist — without one, nothing is advertised
- The DCE side of a serial connection must set the clock rate — identified with `show controllers [interface]`
- The DTE side does not need a clock rate command
